│       → [다중기준선발/20개] → [토큰트리밍]             │
│       → [LLM분류 ⏳ 2~5초] → [Agent판단] → [감성분석] │
└─────────────────────────────────────────────────────────┘
        │  완료 후 다음 영상 시작
        ▼
┌─────────────────────────────────────────────────────────┐
│  Video 2                                                │

총 소요시간: T(V1) + T(V2) + T(V3) + ...

## 2. 제안 구조 (병렬 처리 — ThreadPoolExecutor)

```
[영상 목록: V1, V2, V3, ...]
        │
        ▼
  ThreadPoolExecutor
  max_workers = 3
        │
   ┌────┼────┐
   ▼    ▼    ▼
  [V1] [V2] [V3]   ← 동시 시작
   │    │    │
   │    │    │   각 영상 독립 실행
   ▼    ▼    ▼
[수집][수집][수집]           ← YouTube API (I/O, GIL 해제 → 진짜 병렬)
   │    │    │
   ▼    ▼    ▼
[필터][필터][필터]           ← 규칙 필터 + 점수화 (CPU, 빠름 <50ms)
   │    │    │
   ▼    ▼    ▼
[LLM ][LLM ][LLM ]          ← Groq API (I/O, GIL 해제 → 진짜 병렬)
⏳대기 ⏳대기 ⏳대기          (단, rate limit 주의)
   │    │    │
   ▼    ▼    ▼
[감성][감성][감성]           ← Groq API (I/O, 진짜 병렬)
   │    │    │
   ▼    ▼    ▼
[DB저장][DB저장][DB저장]     ← I/O, 진짜 병렬
   │    │    │
   └────┴────┘
        │
        ▼
   as_completed()
   결과 수집 및 에러 처리

총 소요시간: max(T(V1), T(V2), T(V3)) + 오버헤드
영상 10개 × max_workers=3 기준 = 약 2분 (↓ 약 60%)
```

### 타임라인

```
시간 →  0s        30s       60s
V1:    [━━━━━━━━━━━━━━━━━━]
V2:    [━━━━━━━━━━━━━━━]
V3:    [━━━━━━━━━━━━━━━━━━━━]
V4:                    [━━━━━━━━━━━]  ← V2 완료 후 슬롯 확보
V5:                         [━━━━━━━━━━━━━━]
```

## 3. 장단점 비교

### 순차 처리

| 항목 | 내용 |
|------|------|
| **장점** | 구현 단순, 디버깅 쉬움 |
| **장점** | Groq rate limit 걱정 없음 |
| **장점** | DB 커넥션 1개로 충분 |
| **장점** | 에러 발생 시 어느 영상인지 즉시 파악 |
| **단점** | LLM 대기 중 CPU/네트워크 완전 낭비 |
| **단점** | 영상 수 늘수록 선형적으로 느려짐 |
| **단점** | 단일 영상 오류가 전체 파이프라인 블로킹 가능 |

### 병렬 처리 (ThreadPoolExecutor)

| 항목 | 내용 |
|------|------|
| **장점** | LLM 대기 시간 동안 다른 영상 처리 진행 |
| **장점** | 전체 처리 시간 약 50~70% 단축 |
| **장점** | 단일 영상 실패가 다른 영상에 영향 없음 (`as_completed` 에러 격리) |
| **장점** | 기존 코드 최소 수정으로 적용 가능 |
| **단점** | Groq 무료 플랜 rate limit 초과 위험 (429 에러) |
| **단점** | DB 동시 쓰기 → 커넥션 풀 설정 필요 |
| **단점** | 로그가 여러 영상에서 섞여 출력됨 (디버깅 불편) |
| **단점** | 규칙 필터/점수화(CPU) 구간은 GIL로 실제 병렬 아님 (단, <50ms라 무시 가능) |

## 4. Groq 무료 플랜 Rate Limit 기준

| 제한 | 값 |
|------|----|
| RPM (Requests Per Minute) | 30 |
| TPM (Tokens Per Minute) | 6,000 |
| RPD (Requests Per Day) | 14,400 |

영상 1개당 LLM 호출 약 3~5회(분류 배치 + 감성분석) 기준:
- `max_workers=3` → 분당 최대 9~15 요청 → **안전**
- `max_workers=5` → 분당 최대 15~25 요청 → **경계선**
- `max_workers=10+` → 429 에러 발생 가능

> **테스트 권장값**: `max_workers=3`

## 5. 핵심 고려사항

### DB 커넥션
병렬로 동시에 쓰면 커넥션 경합 발생 가능.  
→ SQLAlchemy 사용 중이면 `pool_size=5` 이상으로 설정 필요.

### 에러 격리
```python
for future in as_completed(futures):
    try:
        result = future.result()
    except Exception as e:
        # V2가 터져도 V1, V3는 계속 진행
        print(f"[ERROR] {futures[future]}: {e}")
```

### 로그 식별
병렬 실행 시 영상 ID를 로그에 항상 포함해야 추적 가능.
```python
print(f"[{video_id}] Step 3: rule filter → {n}개 통과")
```

## 6. 결론

| | 순차 | 병렬 (workers=3) |
|--|------|-----------------|
| 영상 10개 예상 시간 | ~5분 | ~2분 |
| 구현 난이도 | 낮음 | 낮음 (기존 코드 최소 수정) |
| Groq 무료 플랜 안전성 | 안전 | 안전 (workers=3 기준) |
| 프로덕션 전환 시 | 그대로 | workers 수만 조정 |

> 테스트 환경 기준 `ThreadPoolExecutor(max_workers=3)` 적용 권장.  
> 나중에 유료 Groq 플랜으로 전환하면 `max_workers`만 올려주면 됨.
